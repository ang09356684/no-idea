# 14 — 嚴格地點篩選（選縣市/區後不跨區、不跨縣市補資料）

## 背景

目前行程產生 `src/lib/generate.ts` 在篩選地點時，用「三層串接」決定候選池
（`generate.ts:139-144`）：

```ts
const prioritized = [
  ...shuffle(exactMatch),     // 該區 / 該縣市
  ...shuffle(agnostic),       // district === "不限"
  ...shuffle(otherDistrict),  // 其餘（其他區、其他縣市）← 問題所在
];
```

`pickPlaces` 依序從這個池抽 3 個地點。因為 `otherDistrict` 也在池裡，**當該區/該縣市
資料不足時，會自動用「其他區、其他縣市」的資料補滿**——例如選台中時抽到台北餐廳、
選大安區時抽到信義區活動。

## 目標（使用者需求）

1. **選了縣市 + 行政區**：只用「該區」的資料，**該區沒有的就不要從其他區補**
   （含同縣市其他區、其他縣市都不補）。
2. **只選縣市**：只能從「該縣市」補全，**不能跨縣市拉取資料**。
3. **不限**（沒選縣市）：維持現狀，全部資料都可用。

## 已確認的設計決策

- **地點不明的資料「可作補充保留」**（使用者選擇）：
  仍排除「明確在其他區 / 其他縣市」的資料，但**地點真的無法判定**的項目可作為補充出現。
- **電影視為「地點不明」**：電影本來就沒有地址（全台各大影城都有），
  目前 `atmovies` 卻硬塞 `address: "台北市各大影城"`（`atmovies.ts:76`），
  使其被當成「台北」→ 選台中時會消失、選台北時被當成確定在台北。
  需把電影改成真正的「地點不明」，才能讓它在任何縣市/區都能作為補充。
- **同縣市但「區不明」的項目**（如多數台北售票演出 district 仍是「不限」、但地址是台北市）：
  選台北的某區時，視為**補充保留**（它沒被確認在「其他區」，只是區不明）；
  選台中時則因確認在台北而排除。

## 現況資料盤點（實測 `data/combined/all-places.json`）

`district === "不限"` 共 158 筆，來源與地址特性：

| 來源 | 筆數 | address 特性 | 應視為 |
|------|------|--------------|--------|
| atmovies（電影） | 74 | 全部 `"台北市各大影城"`（合成假地址） | **地點不明**（需修正地址） |
| opentix | 36 | `"台北市"` | 確定台北、區不明 |
| tixcraft | 38 | `"台北國際會議中心TICC"` 等台北場館 | 確定台北、區不明 |
| kham / era-ticket | 6 / 4 | `"台北市"` | 確定台北、區不明 |

- 158 筆**地址全都含「支援縣市」字樣**（目前清一色台北），**0 筆**含其他縣市、**0 筆**完全無縣市。
- 另有少量 `district === "其他"` 的他縣市活動（如高雄 culture-music），address 含「高雄」。
  → 這類**必須排除**於台北/台中等選擇之外（屬其他縣市）。

> 結論：不能單純「保留所有 `不限`」當補充，否則選台中會洩漏一堆台北活動。
> 必須**用 address 判定縣市、用 composite district 判定行政區**，只放行「確定在選擇範圍」或
> 「真的判不出地點」的項目。

## 核心判定模型

對每筆 place 取兩個訊號：

- **`addressCity(p.address)`**：地址裡出現的縣市（全台清單，正規化）。回傳支援縣市 key
  （台北/桃園/宜蘭/台中）、或其他縣市名（高雄…）、或 `null`（判不出，如修正後的電影）。
- **`placeDistrictKey(p)`**：若 `p.district` 是複合鍵（`臺北市大安區`）→ 回傳該鍵；
  否則（`不限` / `其他`）→ `null`。

依篩選模式建池（取代現有三層串接）：

| 模式 | 排除（不放行） | tier1（精準，優先） | tier2（補充） |
|------|----------------|---------------------|----------------|
| **不限** | 無 | 全部 | — |
| **縣市 X**（`X-all`） | `addressCity ∉ {X, null}` | `addressCity === X` | `addressCity === null` |
| **行政區 D（在縣市 X）** | `addressCity ∉ {X, null}` 或 `placeDistrictKey ∉ {D, null}` | `placeDistrictKey === D` | 其餘放行者（`placeDistrictKey === null`） |

池 = `[...shuffle(tier1), ...shuffle(tier2)]`，**不再有 otherDistrict 層**。

驗證對照（以資料現況）：
- 選 **台中（city）**：台北售票演出 `addressCity=台北≠台中` → 排除 ✓；電影 `null` → 補充出現 ✓；
  高雄活動 `addressCity=高雄≠台中` → 排除 ✓；台中歌劇院活動 → tier1 ✓。
- 選 **臺中市西屯區**：西屯活動 → tier1；電影 → tier2；台北/桃園/高雄、台中其他區 → 排除 ✓。
- 選 **臺北市大安區**：大安活動 → tier1；台北區不明演出 + 電影 → tier2；信義區、台中、高雄 → 排除 ✓。
- 選 **不限**：全部（含所有電影、各縣市）✓。

  ```ts
  export function addressCity(address: string): string | null { ... }

  // place.district 若為複合鍵則回傳，否則 null（"不限"/"其他" → null）
  export function placeDistrictKey(district: string): string | null {
    return Object.values(CITY_FULL).some((f) => district.startsWith(f) && district.length > f.length)
      ? district : null;
  }
  ```

## 邊界情況

- **其他縣市活動（高雄等）**：`addressCity` 用全台清單判定 → 選支援縣市時排除；選「不限」時仍出現。
- **電影**：修正地址後 `addressCity=null` → 任何選擇都作補充；「不限」時全數出現。
- **台北區不明演出**：選台北縣市/台北任一區 → 補充保留；選台中 → 排除。
- **地址含「街路名」誤判**（如某台北地址出現「桃園街」）：`addressCity` 子字串比對的既有風險，
  機率低，本 plan 不特別處理（沿用現有 `matchesCity` 同等風險），如需可日後改精準比對。
- **行程長度變短**：池變小後 `pickPlaces` 可能湊不滿 3 個（活動/美食不足），會產生較短或較少行程；
  屬預期（使用者要的就是不亂補）。
